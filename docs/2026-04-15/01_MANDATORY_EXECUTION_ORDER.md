# Mandatory Execution Order

상태: 강제 적용
기준일: 2026-04-16

다음 순서를 절대적으로 시행한다.

1. 문서 통합
- `.md` 파일 등 문서 자산을 전부 일자별 폴더 안에 넣는다.
- 날짜 폴더가 그 날짜 작업의 단일 기준점이 된다.
- 날짜 폴더 형식은 `YYYY-MM-DD`로 고정한다.
- 해당 날짜 작업에서 나온 새 문서는 원칙적으로 같은 날짜 폴더 안에 둔다.

2. 역할별 문서 분리
- 공통 파이프라인 문서는 이 문서 하나로 유지한다.
- 각 Codex는 자기 역할 전용 문서를 따로 본다.
- Codex끼리는 가능한 한 서로의 역할 문서를 읽지 않는다.
- 다른 역할 문서를 읽어야 할 사유가 생기면 해당 작업 문서에 이유를 먼저 남긴다.
- 기본 읽기 경계는 다음과 같이 고정한다.
- 메인 프론트엔드 Codex: `01_MANDATORY_EXECUTION_ORDER.md` + `04_MAIN_FRONTEND_CODEX.md`
- 리팩터링 Codex: `01_MANDATORY_EXECUTION_ORDER.md` + `05_REFACTORING_CODEX.md`
- 소스 인덱싱 Codex: `01_MANDATORY_EXECUTION_ORDER.md` + `06_SOURCE_INDEXING_CODEX.md`
- 상태/퀴즈로직 Codex: `01_MANDATORY_EXECUTION_ORDER.md` + `07_STATE_CODEX.md`

3. 리팩토링
- UI, 상태관리, 퀴즈로직, 저장 구조를 분리 가능한 단위로 정리한다.
- 리팩토링은 안드로이드용 퀴즈앱 중심 기준으로 우선순위를 잡는다.
- 메인 에이전트는 프론트엔드와 화면 구조를 직접 맡는다.
- 실제로 안 쓰는 코드가 확인되면 제거한다.
- 안 쓰는지 확인되지 않은 코드는 억지로 빼지 않는다.

4. 로컬 스모크 테스트
- 리팩토링 직후 최소 실행 확인을 먼저 한다.
- 앱 기동, 핵심 화면 진입, 핵심 상호작용 1회 이상을 확인한다.
- 스모크 테스트가 깨진 상태에서는 커밋하지 않는다.

5. 디버깅
- 스모크 테스트에서 드러난 기능 회귀와 실행 오류를 먼저 잡는다.
- 디버깅은 로그, 재현 절차, 수정 결과를 남길 수 있는 형태로 진행한다.

6. 앱 분리
- 퀴즈추가앱은 데스크톱용으로 분리한다.
- 퀴즈앱은 안드로이드용으로 분리한다.
- 우선순위는 퀴즈관리앱이 아니라 안드로이드용 퀴즈앱이다.

7. 프론트엔드와 상태관리/퀴즈로직 분할
- 메인 에이전트는 프론트엔드와 UI 통합을 중심으로 맡는다.
- 안드로이드용 퀴즈앱 흐름이 우선이며, builder 리팩터링은 quiz flow를 방해하지 않는 범위에서만 진행한다.
- 상태 Codex는 상태관리, 퀴즈로직, 저장 구조처럼 UI 바깥 경계가 분명한 영역을 맡는다.
- 리팩터링 Codex는 구조 분해와 결합도 감소에 집중한다.
- 소스 인덱싱 Codex는 symbol map, 코드 탐색 도구, 인덱싱 문서에 집중한다.
- 분할 기준은 화면/UI와 도메인 로직이 서로 독립적으로 교체 가능하도록 잡는다.

8. 퀴즈 저장 방식 정비
- 효율적이고 안정적인 저장 방식을 다운로드받아 적용하거나, 직접 구현하거나, 환경설정으로 고정한다.
- 저장 방식은 모바일 중심 사용 시나리오에서도 안정적으로 동작해야 한다.

9. 메인 작업과 Codex 병렬 진행
- 메인 에이전트는 프론트엔드/통합 작업을 계속 진행한다.
- 리팩터링 Codex, 소스 인덱싱 Codex, 상태 Codex를 병렬로 호출할 수 있다.
- 각 Codex는 자기 전용 문서와 과업만 기준으로 움직인다.
- 메인 에이전트는 각 Codex 결과를 통합하기 전 직접 검토한다.

10. 커밋
- 작업 단위마다 커밋한다.
- 커밋 전에는 최소 스모크 테스트를 통과해야 한다.
- 커밋 메시지는 하위 Codex와 리뷰 단계가 바로 읽을 수 있게 변경 범위를 좁고 명확하게 적는다.

11. Git 반영 및 하위 Codex 협업 지속
- git에 올리기 전후로 하위 Codex와 협업 루프를 유지한다.
- 메인 에이전트는 프론트엔드와 최종 통합 책임을 유지한다.
- 하위 Codex에는 짧은 잡무보다 긴 과업 단위를 맡긴다.
- 하위 Codex에는 전체 대화 로그를 그대로 넘기지 말고, 현재 문서 기준, 관련 파일, 최근 커밋 diff, 과업 목표를 정리해서 넘긴다.
- 토큰 절약을 위해 불필요한 잡음은 줄이되, 과업 수행에 필요한 문맥은 충분히 준다.
- 하위 Codex가 맡지 않은 프론트엔드 정리와 삭제 판단은 메인 에이전트가 직접 한다.

12. 커밋 허가/거절 리뷰
- 메인 에이전트는 자기 작업과 하위 Codex 작업을 따로 검토한다.
- 메인 에이전트는 각 작업 단위에 대해 `커밋 허가` 또는 `커밋 거절`을 명시한다.
- 거절 시 이유와 수정 요구를 구체적으로 남긴다.
- 승인된 작업만 최종 커밋과 push 대상으로 묶는다.
- 필요하면 Codex별로 별도 리뷰 메모를 남긴다.

13. 커밋 심사
- `commit -> commit 내용 심사 -> 승인/거부` 루프를 고정한다.
- 심사는 메인 에이전트 또는 별도 리뷰 단계가 맡는다.
- 구현 담당 하위 Codex와 심사 역할을 혼동하지 않는다.
- 거부 시 이유를 구체적으로 남긴다.
- 구현 담당 하위 Codex가 즉시 수정할 수 있도록 이유를 하위 수준까지 분해한다.
- 하위 Codex가 보낸 메시지는 바로 읽지 않는다.
- 토큰 사용량 절감을 우선한다.

강제 운영 원칙:
- 사용자는 장시간 작업을 허용했다. 최대 5시간 풀 작업도 허용 범위로 간주한다.
- 순서를 바꿔야 할 사유가 생기면 먼저 문서에 예외 사유를 기록한 뒤 진행한다.
- 문서화 없이 구현부터 시작하지 않는다.
- 디버그 산출물보다 실제 배포 동작 확인을 우선한다.
- 퀴즈관리앱보다 안드로이드용 퀴즈앱을 중심에 둔다.
- 토큰 절약과 지능 유지 목적상, 긴 대화 재진술보다 문서와 커밋 diff를 우선 기준으로 쓴다.
- 메인 프론트엔드 작업은 스크린샷 기반으로 퀴즈 앱 화면을 중심에 두고 전반적인 디자인의 통일성과 완성도를 높이는 방향으로 진행한다.
- builder는 공통 파이프라인과 역할 문서 기준으로만 가볍게 다루고, quiz flow를 방해하지 않게 유지한다.
- 소스 인덱싱 결과는 즉시 재사용 가능한 산출물이어야 하며, 다른 Codex가 바로 적용할 수 있어야 한다.
