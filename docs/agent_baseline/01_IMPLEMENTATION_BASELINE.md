# Implementation Baseline

## 목적

이 문서는 현재 앱 구현의 단일 기준 문서다. 기존의 Confirmed Direction, Inferred Intent, 요청 체크리스트, UX 법칙, Nielsen 휴리스틱, 캡처 기반 요구를 한 곳에 통합한다.

## 문서 해석 원칙

- 사용자 텍스트 요청이 최우선이다.
- `C:\Users\박준용\Desktop\quiz_app_capture`의 폴더 경로와 파일 경로도 요구 의미를 가진다.
- 캡처 이미지는 `현재 버그 재현`, `원하는 배치`, `반응형 상태별 요구`를 함께 담은 기준 자료로 본다.
- 구현 후에는 빌드와 스크린샷 검증으로 시행 여부를 확인한다.

## 플랫폼 및 구조 기준

- 스택은 `React + Tauri`를 유지한다.
- 대상 플랫폼은 Windows 데스크톱이다.
- 멀티 OS 창을 계속 띄우지 않고 단일 메인 창 안에서 화면 전환으로 처리한다.
- 앱 크기는 고정형 데스크톱 앱 기준으로 설계한다.
- 고정 크기 안에서 화면이 깨지지 않고 역할이 분명해야 한다.
- 좌표 클릭 자동화 대신 이름 기반 action 호출을 사용한다.

## 제품 방향

- 이 앱은 덱 기반 퀴즈/암기 앱이다.
- 덱 생성, 덱 설정, 덱 상세, Day 상세, 카드 편집, 대량 입력, 학습 유형 선택, 학습 세션이 역할별 화면으로 분리되어야 한다.
- 한 화면은 핵심 기능 하나를 강조해야 한다.
- 값 설정은 메인 개요 화면이 아니라 전용 화면에서 처리해야 한다.
- 질문과 정답은 별도로 저장한다.
- 카드와 덱은 이미지 붙여넣기를 지원한다.
- 대량 입력은 별도 화면의 단일 에디터에서 처리한다.
- 대량 입력 에디터는 블로그 편집기 같은 구분선을 사용한다.
- 구분선으로 나뉜 블록은 홀수 순서 문제, 짝수 순서 정답으로 인식한다.
- 데이터는 자동 저장된다.
- 학습 모드는 `단답형`, `선택형`, `혼합형`을 지원한다.
- 선택형 오답 보기는 같은 덱의 다른 카드 정답에서 가져온다.
- 정답 시 큰 `O`, 오답 시 큰 정답 표시 후 지연시간 뒤 자동 진행한다.
- 학습 완료 후 `오답만 학습`, `전체 다시 학습`이 가능해야 한다.

## 사용자 의도 통합

- 사용자는 한 화면에 많은 역할이 섞이는 것을 싫어한다.
- 사용자는 메인 화면이 브라우저/런처처럼 동작하기를 원한다.
- 사용자는 큰 글자와 읽기 쉬운 표면을 선호한다.
- 사용자는 시각적으로 제품다운 완성도를 원한다.
- 사용자는 반복 학습 흐름에서 마찰이 적고 빠른 다음 행동을 원한다.
- 사용자는 UI 사용법까지 외우게 만드는 구조를 싫어한다.
- 사용자는 숨은 OS 창 로딩 공백을 싫어한다.

### `01/카드내용편집화면/창크기를 늘렸을 때.png`

- 카드 목록은 왼쪽 열에 둔다.

- 큰 창에서는 `카드 목록 / 작성 축 / 미리보기 축`이 동시에 보인다.

### `01/카드내용편집화면/창크기를 줄였을 때/*.png`

- 작은 창에서도 편집 화면과 미리보기 축이 동시에 보여야 한다.
- 작은 창이라고 해서 미리보기를 아래로 밀어 숨기면 안 된다.
- 카드 목록, 작성 축, 미리보기 축은 스크롤과 내부 높이 조절로 처리하되 역할 분리는 유지한다.

### `01/학습시작버튼 눌렀을 때.png`

- 학습 진입 후 하얀 빈창만 보이는 상태는 명확한 버그로 간주한다.
- 학습 시작 후에는 문제 유형 화면 또는 실제 학습 화면이 즉시 보여야 한다.
- 라우트 전환, 스크롤 위치, 로딩 상태, 런타임 예외로 인해 빈 표면만 보이면 안 된다.

## UI/UX 원칙 통합

### Fitts의 법칙

- 자주 누르는 버튼은 충분히 크게 둔다.
- 학습 완료 행동은 큰 버튼 2개로 제한한다.
- 객관식 선택 버튼은 큰 타깃을 유지한다.

### Gestalt의 법칙

- 덱, Day, 카드, 학습 세션은 기능 단위 패널로 분리한다.
- 관련 정보는 같은 카드군으로 묶는다.
- 한 화면의 핵심 행동과 보조 행동은 시각적 위계를 분리한다.

### Miller의 법칙

- 한 번에 보여주는 핵심 선택지는 적게 유지한다.
- 완료 후 다음 행동은 2개 수준으로 제한한다.
- 메인 화면은 핵심 액션 위주로 정리한다.

### Progressive Disclosure

- 대량 입력은 단일 카드 편집 화면과 분리한다.
- 도움말은 별도 화면 대신 맥락 내부의 짧은 힌트로 넣는다.
- 고급 정보는 기본 작업을 가리지 않아야 한다.

### F 패턴 / Z 패턴

- 목록 화면은 빠르게 훑을 수 있어야 한다.
- 시작/완료 화면은 핵심 행동으로 시선이 자연스럽게 이어져야 한다.

### Aesthetic-Usability Effect

- 화면은 기능만 되는 수준이 아니라 상용 앱처럼 정돈되어야 한다.
- 미려함은 이해 속도와 사용 신뢰도를 높이는 요소로 본다.

### Jacob의 법칙

- 목록, 폼, 저장, 뒤로가기, 완료 흐름은 익숙한 데스크톱 패턴을 따른다.
- 단축키는 일반적인 `Ctrl+N`, `Ctrl+S`, 숫자키, `Enter` 규칙을 우선한다.

### Nielsen 10 휴리스틱 통합

- 시스템 상태 표시: 저장/로딩/오류를 보여준다.
- 실제 세계 언어: 한국어 중심 문구를 사용한다.
- 사용자 제어: 뒤로, 목록으로, 취소를 유지한다.
- 일관성: 공통 Shell과 버튼 계층을 유지한다.
- 오류 예방: 빈 입력, 삭제, 불완전 대량 입력을 사전에 막는다.
- 기억보다 인식: 단축키와 현재 상태를 눈에 보이게 둔다.
- 유연성과 효율성: 마우스와 키보드 흐름을 둘 다 지원한다.
- 미니멀 디자인: 핵심 액션을 가리지 않는 정보만 둔다.
- 명확한 오류 메시지: 무엇을 고쳐야 하는지 바로 보여준다.
- 도움말 제공: 별도 도움말 화면 대신 현재 맥락에서 짧게 제공한다.
