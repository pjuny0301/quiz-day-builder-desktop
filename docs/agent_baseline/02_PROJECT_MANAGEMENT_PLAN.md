# 프로젝트 관리 계획서

## 1. 프로젝트 개요 및 목표

### 1.1 문서 목적 및 범위

본 문서는 `Quiz Day Builder Desktop` 프로젝트의 관리 기준 문서이다.  
문서 목적은 다음과 같다.

- 프로젝트의 범위와 목표를 명확히 정의한다.
- 개발 절차, 일정, 역할, 품질 관리 기준을 통합 관리한다.
- 개인 프로젝트이지만 역할을 구분하여 실행과 검증 책임을 명확히 한다.
- 향후 기능 확장 및 유지보수 시 기준 문서로 활용한다.

적용 범위는 다음과 같다.

- Windows 데스크톱용 퀴즈/암기 앱 개발
- React + Tauri 기반 단일 메인 창 UI
- 덱/Day/카드 관리, 학습 세션, 자동 저장, 배포 파일 생성
- 프로젝트 산출물, 품질 관리, 변경 관리, 리스크 관리

### 1.2 프로젝트 개요

| 항목 | 내용 |
|---|---|
| 프로젝트명 | Quiz Day Builder Desktop |
| 버전 기준 | 0.1.0 |
| 개발 형태 | 개인 프로젝트 |
| 대상 플랫폼 | Windows Desktop |
| 주요 기술 | React, TypeScript, Vite, Tauri, Rust |
| 저장 방식 | JSON 기반 로컬 저장 |
| 핵심 개념 | Deck, Day, Card, Study Session |

본 프로젝트는 사용자가 암기용 덱을 만들고, 카드를 `Day` 단위로 자동 분할하여 학습할 수 있는 데스크톱 앱을 구축하는 것을 목표로 한다.  
기존 단순 유틸리티형 구조가 아니라, 한 화면에 한 핵심 기능만 두는 상용 앱형 UX를 지향한다.

### 1.3 프로젝트 목표

#### 기능 목표

- 덱 생성, 수정, 삭제 기능 제공
- 카드 문제/정답 분리 저장
- 카드 수 기준 Day 자동 분할
- 빈 덱이어도 `Day 1` 기본 생성
- 이미지 붙여넣기 기반 카드 작성 지원
- 단답형, 선택형, 혼합형 학습 지원
- 오답 재학습 및 전체 재학습 지원
- 자동 저장 및 Windows 실행 파일 배포

#### 품질 목표

- 단일 메인 창 기반의 빠른 화면 전환
- 화면별 역할 분리
- 사용자가 직관적으로 이해할 수 있는 한국어 UI
- 읽기 쉬운 텍스트와 단순한 정보 구조 유지
- 기능 추가 시 구조가 무너지지 않도록 확장성 중심 리팩터링 유지

## 2. 개발 계획

### 2.1 개발 절차 모형

본 프로젝트는 `반복적 점진 개발(Iterative + Incremental)` 모형을 적용한다.

적용 이유는 다음과 같다.

- 사용자 요구가 개발 중 자주 구체화되고 수정되었다.
- UI/UX 요구가 기능 구현과 동시에 계속 조정되었다.
- 개인 프로젝트 특성상 문서-구현-검증 사이클을 짧게 반복하는 방식이 효율적이다.

개발 절차는 다음 순서로 반복한다.

1. 요구사항 문서 갱신
2. 화면/상태/저장 구조 검토
3. 기능 구현
4. 빌드 및 실행 검증
5. UX 보정 및 리팩터링

## 3. 팀 구성 및 역할

본 프로젝트는 개인 프로젝트로 수행되지만, 역할을 다음과 같이 구분하여 관리한다.

| 역할 | 인원 | 주요 책임 | 주요 산출물 |
|---|---:|---|---|
| PM | 1 | 범위 관리, 일정 관리, 문서 승인, 우선순위 조정 | 계획서, 일정표, 변경 이력 |
| 요구사항 분석 | 1 | 요구사항 정리, 사용 시나리오 정리, 참조 문서 갱신 | 방향 문서, 유스케이스 |
| 프런트엔드 개발 | 1 | React 화면 구현, UX 개선, 상태 연결 | 화면 컴포넌트, 스타일 |
| 데스크톱/백엔드 개발 | 1 | Tauri 라우팅, 저장 처리, 빌드 환경 관리 | Rust 저장 로직, 빌드 설정 |
| 테스트/품질 담당 | 1 | 빌드 검증, 오류 재현, 체크리스트 관리 | 테스트 결과, 이슈 목록 |
| 형상/배포 담당 | 1 | 산출물 정리, 설치 파일 생성, 버전 관리 | exe, msi, setup.exe |

### 3.1 역할별 책임 구분 원칙

- PM 역할은 기능 추가 여부와 범위 변경을 승인한다.
- 개발 역할은 코드 구조와 구현 품질을 책임진다.
- 테스트 역할은 기능 정상 동작과 회귀 여부를 검증한다.
- 배포 역할은 최종 산출물의 버전과 경로를 관리한다.

## 4. 품질 관리 및 운영 계획

### 4.1 팀 미팅 계획

개인 프로젝트이므로 형식적 회의 대신 정기 점검 체계를 운영한다.

| 구분 | 주기 | 목적 | 결과물 |
|---|---|---|---|

| 기능 완료 리뷰 | 기능 단위 | 구현/문서/빌드 동시 확인 | 완료 체크 기록 |
| 배포 전 점검 | 배포 직전 | 산출물 확인, 설치 파일 점검 | 배포 체크리스트 |

### 4.2 변경사항 관리

변경사항은 다음 절차를 따른다.

1. 요구 변경 발생
2. 영향 범위 분석
3. 관련 문서 우선 수정
4. 코드 수정 및 리팩터링
5. 빌드와 실행 확인
6. 산출물 및 문서 버전 갱신

변경사항 관리 원칙은 다음과 같다.

- 문서보다 코드가 먼저 바뀌지 않도록 한다.
- 화면 역할이 바뀌면 반드시 유스케이스와 방향 문서를 함께 갱신한다.
- 급한 수정도 함수 주석과 구조 정리를 생략하지 않는다.
